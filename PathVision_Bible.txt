CHAPTER 1: PROJECT OVERVIEW
-----------------------------------------------------------------------------
1.1 INTRODUCTION
PathVision is a local-first orientation and navigation assistant designed for visually impaired individuals. Running entirely on local laptop hardware, PathVision uses machine learning models to detect walkable paths and obstacles, resolving navigation directions in milliseconds and communicating descriptions naturally via synthetic speech.

1.2 PROBLEM STATEMENT
Visually impaired users face significant risks when navigating unfamiliar spaces. Canes offer short-range checks but miss low-hanging obstacles. GPS fails indoors, and cloud-based vision assistants incur significant latency (often >1.5 seconds) due to round-trip times, creating safety risks during active walking.

1.3 OBJECTIVES & DESIGN PHILOSOPHY
* Local-First Execution: No cloud API calls in the critical path to ensure privacy and low latency.
* Real-Time Vision: Maintain a steady 30 FPS pipeline on budget GPUs (such as RTX 2050).
* Cognitive Comfort: Prioritize quiet navigation, speaking natural cues only when environmental changes occur.

1.4 TARGET USERS & USE CASES
Designed for white cane users who need higher-level context. Key use cases include: indoor office corridors, campus hallways, transition gateways, and quiet indoor spaces where safe walkable path boundaries are crucial.

CHAPTER 2: REPOSITORY STRUCTURE
-----------------------------------------------------------------------------
The PathVision repository is structured into isolated layers separating perception wrappers, safety planning modules, and asynchronous output workers. Below is the file structure layout:

* perception/
  - pathvision_trt.py: Safe path segmenter TensorRT wrappers.
  - depth_trt.py: Depth Anything V2 monocular depth wrapper.
  - scene_fusion.py: Fuses mask + depth summaries atomically.
  - navigation_mesh.py: Generates the boundary corridor node graph.
* navigation/
  - decision.py: Calculates steering curves and deadbands.
  - safety.py: Evaluates danger thresholds (SAFE/CAUTION/DANGER).
  - path_geometry.py: Analyzes shape boundaries and centroids.
* reasoning/
  - qwen_llama.py: Offline LLM reasoner utilizing llama.cpp.
  - prompts.py: System and mode-specific chat message layouts.
  - conversation_memory.py: Throttling loops and reassurance.
  - situation_manager.py: Environment situation classifiers.
  - scene_memory.py: In-memory sliding window history logs.
* speech/
  - kokoro.py: Priority-based style voice synthesis playback.
* runtime/
  - runtime_manager.py: Lifecycle state machine, watchdogs, and boot orchestrator.
  - runtime_state.py: Boot and run status definitions.
  - event_bus.py: Atomic publish/subscribe message router.
  - world_model.py: Thread-safe repository of current state.
  - scheduler.py: Orchestrates 30 FPS visual processing loop.
  - health_monitor.py: Watches loop timers and resets hung workers.
* learning/
  - dataset_builder.py: Telemetry loops writing hard samples for retraining.
  - auto_label.py: Triggers for identifying hard test scenes.
  - feedback.py: Logs keyboard hotkeys pressed by operators.
  - scene_logger.py: Writes compact telemetry metadata.
* config.py: Configuration loading from environment variables.
* main.py: Execution entry point and window manager.

CHAPTER 3: COMPLETE SOFTWARE ARCHITECTURE
-----------------------------------------------------------------------------
PathVision uses a highly concurrency-isolated, multi-threaded design. The critical vision loop runs continuously without blocking, while cognitive reasoning and voice synthesis are deferred to asynchronous worker threads.

3.1 CONCURRENCY MODEL AND THREADS
Execution is divided across five specialized threads to prevent latency bottlenecks:
* Main GUI Thread: Polling keyboard listener hotkeys and rendering OpenCV previews.
* FastVisionPipeline Thread (Scheduler): Capturing frame and triggering segmenter/depth GPU streams.
* EventBus Thread: Managing priority events and routing triggers synchronously to subscribers.
* QwenReasoner Thread: Querying the local llama.cpp instance for scene descriptions.
* SpeechWorker Thread: Directing synthetic waveform playback in single-threaded audio loops.

3.2 DATA FLOW SCHEMA
[Camera] -> [Scheduler] -> [TRT Engines] -> [SceneFusion] -> [WorldModel] -> [EventBus] -> [Reasoner] -> [Speech]

CHAPTER 5: LIBRARIES & DEPENDENCY ENCYCLOPEDIA

To ensure reliable local performance, third-party libraries were carefully audited for thread safety and memory leakage under Windows:

* TensorRT (Inference Compiler, GPU): Minimizes kernel launch overhead via compiled graph.
* PyTorch (Array Operations, GPU/CPU): Leverages CUDA streams for in-place GPU tensor decoding.

* sounddevice (Audio Playback, CPU): Direct PortAudio bindings. Required custom polling loop.
* OpenCV (Image Processing, CPU): Highly optimized contour extraction and Connected Components.

CHAPTER 6: AI MODELS & RUNTIME ASSETS
-----------------------------------------------------------------------------
The runtime operates four localized AI model weights, compiled or quantized for budget hardware target constraints:

* PathVision segmenter: input (1, 3, 240, 320), output (1, 3, 240, 320) class logs, FP16, VRAM: 15 MB.
* Depth Anything V2: input (1, 3, 518, 518), output (1, 518, 518) depth map, FP16, VRAM: 98 MB.
* Qwen-2.5-VL-3B: context: 2048, Int4 (Q4_K_M GGUF), RAM: 2100 MB (CPU offloaded).
* Kokoro-82M: style synthesis, FP32, VRAM: 180 MB.


-----------------------------------------------------------------------------
CHAPTER 7: COMPUTER VISION PIPELINE
-----------------------------------------------------------------------------
7.1 PIPELINE EXECUTION FLOW
1. Camera Capture: Async camera thread fetches raw BGR frame from webcam buffer.
2. Preprocessing: FramePreprocessor resizes image, converts to RGB, and normalizes channels in-place on pinned host memory.
3. Dual Inference: Preprocessed frames are copied to GPU asynchronously and execute simultaneously in stream cuda_stream_high.
4. Decoupled Decoding: SegmentationDecoder computes softmax and argmax on GPU, transferring only the final mask back.
5. Depth Normalization: Depth engine performs in-place min/max stretching to yield distance values in [0.0, 1.0].
6. Scene Fusion: Combines boundaries and clearance measurements into a unified ScenePacket.


-----------------------------------------------------------------------------
CHAPTER 8: NAVIGATION ENGINE
-----------------------------------------------------------------------------
8.1 HYSTERESIS PLANNING
To prevent command chatter (rapid switching between steering commands), the engine implements a state hysteresis filter. Safety cleared states must remain stable for 3 consecutive frames to block, and 2 frames to unblock, smoothing fluctuations from lighting changes.

8.2 CURVE ANTICIPATION
When centerline coordinates are extracted from the NavigationMesh, the decision engine applies weighted averaging. Higher weights are assigned to look-ahead points at the top of the corridor, helping the user anticipate curves before they reach them.


-----------------------------------------------------------------------------
CHAPTER 9: DEPTH ESTIMATION SYSTEM
-----------------------------------------------------------------------------
Monocular depth estimation is performed using Depth Anything V2. The output represents relative depth (0 = near, 1 = far).

9.1 CORRIDOR MASK FILTERING
To prevent false warnings from nearby side walls or the floor directly at the user's feet, the nearest obstacle distance calculation is masked. The system computes the 10th percentile of depth values only within the pixels marked as 'walkable safe path'. If no path is visible, it falls back to the full frame.


-----------------------------------------------------------------------------
CHAPTER 10: OBJECT DETECTION PIPELINE
-----------------------------------------------------------------------------
Object detection is planned as a future integration (e.g. YOLOv10-Seg) to label obstacles. Currently, the system relies strictly on monocular depth clearance, treating any near structures in the walkable path as generic hazards, which simplifies VRAM allocation and minimizes processing latency.


-----------------------------------------------------------------------------
CHAPTER 11: SCENE REASONING ENGINE
-----------------------------------------------------------------------------
The reasoning engine leverages the local Qwen LLM. Instead of issuing commands, it constructs user prompts with environment parameters (safe area ratio, closest obstacle distance, confidence) and historical context, generating natural walking descriptions.

11.1 CONCURRENCY SAFEGUARDS
LLM text generation can take up to 2 seconds. To prevent blocking the 30 FPS vision pipeline, reasoning calls run on a separate thread using a single-flight mutex lock. If a generation call is still active when a new one is requested, the request is dropped to prevent thread leaks.


-----------------------------------------------------------------------------
CHAPTER 12: SPEECH & AUDIO PIPELINE
-----------------------------------------------------------------------------
Voice playback uses the KokoroSpeaker worker thread. The synthesizer converts prompt text into audio waveforms using a priority queue.

12.1 PREEMPTION AND PRE-EMPTION POLLING
High-priority safety warnings (Priority 0/1) immediately flush the playback queue. To prevent PortAudio driver access violations, the background thread checks a preemption flag every 50ms inside a loop rather than stopping the sounddevice stream directly from other threads, keeping all driver interactions single-threaded.


-----------------------------------------------------------------------------
CHAPTER 13: RUNTIME OPTIMIZATION
-----------------------------------------------------------------------------
Key performance optimizations implemented in PathVision include:
* FP16 TensorRT Compilation: Compiling segmenter and depth models to half precision, cutting inference latencies by 60%.
* Pinned Memory Allocation: Allocating non-pageable pinned host buffers, increasing host-to-device copy bandwidth.
* In-Place NumPy Math: Pre-allocating processing buffers and using in-place operations to avoid memory allocation delays.

CHAPTER 16: IMPROVEMENT HISTORY
-----------------------------------------------------------------------------
Key improvements made during the v2.0 hardening phase include:
* GPU-Based Logits Decoding: Moving argmax and softmax operations from the CPU to PyTorch GPU tensors, saving 4.5ms per frame.
* PortAudio Crash Patch: Replacing direct multi-threaded stop calls with a 50ms polling loop, eliminating audio driver freezes.
* Depth Masking: Quantizing depth maps only within the walkable path, resolving false danger triggers from side walls.


-----------------------------------------------------------------------------
CHAPTER 17: ENGINEERING DECISIONS & TRADE-OFFS
-----------------------------------------------------------------------------
17.1 LOCAL CPU LLM OFFLOADING
* Decision: Offload Qwen-2.5-VL layers to the CPU via llama.cpp instead of the GPU.
* Trade-off: CPU inference is slower (20 t/s vs 60 t/s), but it frees up GPU VRAM for the 30 FPS vision engines, preventing out-of-memory crashes on 4GB VRAM cards.

17.2 PREEMPTIVE POLLING VS DIRECT STOPS
* Decision: Use a 50ms polling loop for audio preemption instead of direct sounddevice thread stops.
* Trade-off: Introduces a minor 50ms delay in safety alerts, but prevents PortAudio driver access violations, ensuring system stability.

CHAPTER 18: ALGORITHMS & MATHEMATICAL FOUNDATIONS
-----------------------------------------------------------------------------
18.1 CONNECTED COMPONENTS EXTRACTION
The system filters the segmentation mask by identifying the largest component that touches the bottom row of the frame, representing the ground at the user's feet:

```python
# Find connected components
num_labels, labels, stats, centroids = cv2.connectedComponentsWithStats(mask_u8, connectivity=8)

# Find largest component touching the bottom row
best_label = -1
best_area = -1
for label in range(1, num_labels):
    x, y, w, h, area = stats[label]
    touches_bottom = (y + h) >= (model_h - 1)
    if touches_bottom and area > best_area:
        best_area = area
        best_label = label
```

18.2 CENTERLINE OFFSET STEERING
The steering target is calculated by evaluating the horizontal offset of the path centerline from the image center:
Offset = Steer_x - (Frame_Width / 2)
If |Offset| <= Deadband_px, the path is considered centered. Otherwise, the system recommends steering in the direction of the offset.

CHAPTER 21: THREADING, MEMORY & RUNTIME ARCHITECTURE
-----------------------------------------------------------------------------
To maintain high performance, the system allocates resources deterministically during startup and runs continuous checks:

21.1 STAGED BOOT SEQUENCE
The boot sequence proceeds through six distinct states: BOOTING (initializing events), DISCOVERING_HARDWARE (scanning cameras), LOADING_MODELS (allocating GPU memory and loading TensorRT engines), WARMING_UP (running dummy runs to allocate driver contexts), READY, and NAVIGATING (starting vision and reasoning loops).

21.2 HEALTHMONITOR WATCHDOG
The HealthMonitor thread checks for worker heartbeats. If a pipeline loop hangs (e.g., due to camera driver timeout), the monitor logs a warning and restarts the worker thread automatically, recovering the system without user intervention.


-----------------------------------------------------------------------------
CHAPTER 22: ERROR HANDLING, LOGGING & DEBUGGING
-----------------------------------------------------------------------------
The system is designed to handle hardware and model failures gracefully, prioritizing safety:
* Camera Disconnection: If the camera drops frames, the scheduler uses the last cached frame for up to 3 frames before triggering an emergency STOP.
* LLM Timeout: If a Qwen generation call exceeds 4 seconds, the reasoning worker cancels it and uses a deterministic fallback instruction based on current path geometries.
* JSONL Log Protection: All file writes are wrapped in try-except blocks, ensuring disk write failures do not disrupt the active navigation loop.
